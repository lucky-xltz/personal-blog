# Rust重写成TypeScript反而快了3倍：WASM边界税与重写幻觉的深度剖析

OpenUI团队最近发布了一篇令人意外的技术博客：他们将一个用Rust编写、编译为WASM的解析器完全重写为TypeScript后，**单次调用快了2.2-4.6倍，流式处理总成本降低了2.6-3.3倍**。

这组数据在技术社区引发了激烈讨论。Rust不是应该比JavaScript快吗？WASM不是号称"接近原生性能"吗？到底哪里出了问题？

答案藏在一个大多数人忽略的地方——**WASM与JavaScript之间的边界税**。

## 一个六阶段解析器的故事

OpenUI的`openui-lang`解析器负责将LLM生成的自定义DSL转换为React组件树。它的管线有六个阶段：

```
autocloser → lexer → splitter → parser → resolver → mapper → ParseResult
```

这个解析器在每个流式chunk上调用——LLM每吐出20个字符就要触发一次解析。延迟直接决定用户体验。

最初选择Rust+WASM的理由很充分：Rust快，WASM给浏览器带来接近原生的速度，解析器本身也是计算密集型任务。逻辑上没毛病。

但实际测量后发现，**瓶颈根本不在Rust代码本身**。

## WASM边界税：被忽视的隐性成本

每次调用WASM解析器，都要支付一笔固定的"过路费"。流程是这样的：

```
JavaScript 堆                          WASM 线性内存
────────────────────────────────────────────────────
wasmParse(input)
  │
  ├─ 复制字符串: JS堆 → WASM线性内存    (分配 + memcpy)
  │
  │                              Rust解析执行    ✓ 本身很快
  │                              serde_json::to_string()  ← 序列化结果
  │
  ├─ 复制JSON字符串: WASM → JS堆        (分配 + memcpy)
  │
  JSON.parse(jsonString)               ← V8反序列化
  │
  return ParseResult
```

看到了吗？**Rust解析本身从来不是瓶颈**。开销全部来自边界：复制字符串进来，把结果序列化为JSON字符串，复制JSON字符串出去，然后V8再把它反序列化回JS对象。

这不是微不足道的开销。对于一个被调用50次的流式场景，每次边界穿越的固定成本累加起来足以淹没Rust的计算优势。

## serde-wasm-bindgen：好心办坏事

团队的第一个反应是：既然JSON往返太贵，能不能让WASM直接返回JS对象？`serde-wasm-bindgen`就是干这个的——它把Rust结构体直接转换为`JsValue`返回。

结果呢？**比JSON方案还慢了9-29%**。

| 测试用例 | JSON往返 | serde-wasm-bindgen | 变化 |
|---------|---------|-------------------|------|
| simple-table | 20.5µs | 22.5µs | 慢9% |
| contact-form | 61.4µs | 79.4µs | 慢29% |
| dashboard | 57.9µs | 74.0µs | 慢28% |

为什么会这样？因为**JS无法直接读取Rust结构体在WASM线性内存中的字节**——两个运行时使用完全不同的内存布局。`serde-wasm-bindgen`必须递归地将Rust数据逐字段构造为真实的JS数组和对象，这意味着**大量细粒度的跨边界转换**。

对比JSON方案：`serde_json::to_string()`在纯Rust中运行（零边界穿越），产生一个字符串，一次memcpy复制到JS堆，然后V8原生的C++ `JSON.parse`用一个优化过的pass处理完毕。

**更少、更大、更优化的操作，胜过大量细小操作。**

## 直接去掉WASM：TypeScript的胜利

团队做了最直接的决定：把整个六阶段管线移植到TypeScript。同样的架构，同样的输出格式，没有WASM，没有边界，完全在V8堆内运行。

| 测试用例 | TypeScript | WASM (JSON往返) | 加速比 |
|---------|-----------|----------------|-------|
| simple-table | 9.3µs | 20.5µs | 2.2倍 |
| contact-form | 13.4µs | 61.4µs | 4.6倍 |
| dashboard | 19.4µs | 57.9µs | 3.0倍 |

TypeScript快了3-5倍。不是因为TypeScript比Rust快——**V8的JIT编译器消除了Rust在小规模解析任务上的任何计算优势**，而边界开销被彻底移除。

## 真正的性能杀手：O(N²)流式架构

消除WASM后，团队发现了更深层的问题。原始架构是这样的：

```
Chunk 1:  parse("root = Root([t")              → 14字符
Chunk 2:  parse("root = Root([tbl])\ntbl = T") → 27字符
Chunk 3:  parse(full_accumulated_string)        → ...
```

每个chunk到来时，**从头重新解析整个累积字符串**。对于1000字符的输出、20字符的chunk：50次parse调用，累计处理约25,000字符。**O(N²)复杂度**。

关键洞察：**以深度0换行结束的语句是不可变的**——LLM不会回头修改已完成的语句。团队实现了增量解析器，缓存已完成语句的AST：

```typescript
interface StreamingState {
  buf: string;
  completedEnd: number;      // 已缓存到哪个位置
  completedSyms: Map<string, AST>;  // 已完成语句的AST缓存
  firstId: string | null;
}

function push(chunk: string): ParseResult {
  // 1. 从completedEnd开始扫描深度0换行
  // 2. 对每个找到的完整语句：解析+缓存AST → 推进completedEnd
  // 3. 末尾未完成的语句：自动关闭+重新解析
  // 4. 合并缓存+待处理 → resolve → map → 返回ParseResult
}
```

已完成的语句不再被重新解析。每个chunk只重新解析尾部的未完成语句。**从O(N²)降到O(N)**。

| 测试用例 | 朴素重解析 | 增量缓存 | 加速比 |
|---------|----------|---------|-------|
| simple-table | 697µs | 697µs | 无（单语句，无缓存收益） |
| contact-form | 316µs | 122µs | 2.6倍 |
| dashboard | 840µs | 255µs | 3.3倍 |

注意`simpl e-table`没有提升——它是单语句，没有可缓存的内容。收益随语句数量增长而增长。

## "重写幻觉"：为什么重写几乎总是更快

HN评论区的一条高赞评论切中要害：

> "我们把代码从语言L重写到语言M，结果更快了！"这很正常——你有机会纠正所有纠缠和扭曲，避开每一个已知的坏决策，应用新发明的更好方法。**即使L = M（同语言重写），这个结论也成立。**加速来自重写和重新思考，而非语言本身。

这就是**重写幻觉**（Rewrite Bias）。当你重写代码时：

1. **你带着所有教训重来**——第一次实现中的所有设计错误都被修正
2. **你应用了新知识**——重写时你知道了更好的算法和数据结构
3. **你简化了架构**——积累的复杂性被清除
4. **你测量了性能**——原始实现往往从未被仔细测量

OpenUI的案例完美体现了这一点：他们不仅去掉了WASM，还顺便修复了O(N²)的流式架构。如果他们在Rust中也实现增量解析，差距会小很多。

## WASM什么时候才值得用？

这次经历让团队（和社区）重新审视WASM的适用场景：

### ✅ 适合WASM的场景

**计算密集、边界交互最少的任务**：图像/视频处理、密码学运算、物理模拟、音频编解码器。大输入 → 标量输出或原地修改。边界很少被穿越。

**移植现有C/C++库**：把SQLite、OpenCV、libpng等原生库搬到浏览器，无需完整JS重写。这是WASM最大的价值所在。

**长时间运行的纯计算**：在Web Worker中运行的后台计算，不频繁与主线程交互。

### ❌ 不适合WASM的场景

**将结构化文本解析为JS对象**：无论如何都要付出序列化代价。解析计算本身足够快，V8的JIT消除了Rust的任何优势。边界开销占主导。

**频繁调用的小输入函数**：如果函数每秒调用50次，计算只需5µs，边界成本无法被摊薄。

**任何需要频繁往返JS的数据转换**：每次"把数据从WASM世界搬到JS世界"都有成本，累积起来很可观。

## 实践指南：如何选择WASM还是原生JS/TS

决策流程很简单：

```
1. 你的瓶颈是计算密集型吗？
   │
   ├─ 否 → 用JS/TS，别想WASM
   │
   ├─ 是 → 计算需要频繁与JS交互吗？
   │        │
   │        ├─ 是 → 用JS/TS，或重构架构减少交互
   │        │
   │        └─ 否 → 用WASM，但要：
   │                 - 最小化边界数据量
   │                 - 优先使用TypedArray/SharedArrayBuffer
   │                 - 避免JSON往返，考虑二进制协议
   │
   2. 你有现成的C/C++/Rust库需要移植？
      │
      ├─ 是 → WASM是正确选择
      │
      └─ 否 → 先用JS/TS实现，性能不足再考虑WASM
```

## WASM与JS不共享堆内存

这一点值得单独强调：**WASM和JavaScript使用完全不同的内存模型**。

WASM有平坦的线性内存（一块连续的`ArrayBuffer`），而V8管理的是带GC的堆，对象有隐藏类、内联缓存、指针压缩等。两者之间的数据传输本质上是**跨进程通信**的简化版——你需要序列化、复制、反序列化。

这意味着：如果你的WASM模块需要频繁返回复杂嵌套的JS对象，你可能在支付比纯JS实现更高的总成本。

更好的模式是：让WASM操作`TypedArray`或`SharedArrayBuffer`中的原始数据，JS端直接读取这些共享缓冲区，避免任何序列化开销。

## V8的JIT有多强？

一个常被低估的事实：**V8的TurboFan JIT编译器对JavaScript的优化程度已经非常高**。

对于解析器这类任务，V8能做到：
- 字符串操作被优化为直接内存操作
- 热路径函数被编译为原生机器码
- 内联缓存消除了大部分动态类型开销
- 隐藏类系统让对象访问接近C结构体的性能

当你的Rust代码需要穿越WASM边界时，它丧失了这些优化带来的连贯性。V8无法对跨越边界的调用链做整体优化——它只能在各自的运行时内分别优化。

## 总结

OpenUI的案例给所有考虑WASM的团队上了三堂课：

**第一课：先测量，再优化。** 他们的Rust代码本身很快，但瓶颈在边界。不测量就不知道真正的瓶颈在哪里。

**第二课：算法改进 > 语言选择。** 从O(N²)到O(N)的流式缓存带来的提升，比从WASM到TypeScript的切换更大。选对算法比选对语言重要一个数量级。

**第三课：警惕重写幻觉。** 重写几乎总是更快，因为你在修正设计错误。如果要在两种语言间做比较，请确保算法和架构相同——否则你比较的是"旧代码 vs 新代码"，而非"语言A vs 语言B"。

WASM不是银弹。它在正确的场景下（计算密集、低频交互、移植原生库）是强大的工具。但把它用在错误的地方——比如频繁的结构化数据解析——你可能在为一笔不存在的性能收益付出实实在在的边界税。

**在选择实现语言之前，先搞清楚时间到底花在了哪里。**

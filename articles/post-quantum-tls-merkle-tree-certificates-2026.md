# 量子计算机还没来，TLS 证书已经要换了

2026 年 6 月 3 日，Let's Encrypt 发布了一篇看似技术性、实则影响深远的博文：宣布将采用 **Merkle Tree Certificates (MTCs)** 作为其后量子时代的证书方案，目标是 2026 年底上线测试环境，2027 年进入生产。

这条消息在 Hacker News 上引发了 264 分的热议。为什么？因为这是全球最大的证书颁发机构（签发了超过一半的 HTTPS 证书）在告诉我们：**后量子迁移不再是学术讨论，而是正在执行的工程计划**。

本文将从密码学原理、工程约束、架构设计三个层面，深入解析这场正在发生的 TLS 基础设施变革。

## 量子威胁：不只是"加密"的问题

过去几年，后量子密码学的讨论主要聚焦在**加密**（confidentiality）上。逻辑很简单：攻击者今天录制加密流量，未来量子计算机破解密钥，就能解密历史数据——这就是所谓的 **"先存储，后解密"（harvest now, decrypt later）** 攻击。

但 Let's Encrypt 的文章指出了一个更紧迫的问题：**认证**（authentication）同样面临量子威胁。

当前的 TLS 握手依赖非对称密码学来验证服务器身份：

```
客户端 → 服务器：ClientHello
服务器 → 客户端：证书（含 RSA/ECDSA 公钥 + CA 签名）
客户端验证：用 CA 的公钥验证证书签名 ✓
```

如果量子计算机能破解 RSA 或 ECDSA，攻击者可以**伪造证书签名**，实施中间人攻击。这比"解密历史流量"更危险——它意味着**实时的、主动的攻击**。

### 为什么"长寿命密钥"是最大风险

并非所有密钥面临的风险相同。文章特别强调了**长寿命密钥**（long-lived keys）的脆弱性：

| 密钥类型 | 典型寿命 | 量子威胁等级 |
|---------|---------|------------|
| TLS 服务器证书 | 90 天 | 中等（频繁轮换） |
| 根 CA 证书 | 10-20 年 | **极高** |
| 代码签名密钥 | 3-5 年 | 高 |
| 身份/身份验证系统 | 5-10 年 | 高 |

根 CA 证书是整个 Web PKI 信任链的锚点。如果攻击者在量子计算机可用后伪造根 CA 签名，整个 HTTPS 生态系统将面临系统性风险。由于新密码学技术从标准化到广泛采用通常需要 **5-10 年**，现在启动迁移并不算早。

### 各方时间表

| 组织 | 行动 | 时间线 |
|------|------|--------|
| 美国 NSA (CNSA 2.0) | 指导国家安全系统迁移 | 2030-2035 |

## 后量子签名的"尺寸炸弹"

NIST 在 2024 年标准化了基于格（lattice-based）的签名算法 **ML-DSA**（Module-Lattice-Based Digital Signature Algorithm）。但问题来了——后量子签名的尺寸远大于传统算法：

```
签名尺寸对比：
┌─────────────────┬──────────┬──────────────────┐
│ 算法             │ 签名大小  │ 公钥大小          │
├─────────────────┼──────────┼──────────────────┤
│ ECDSA-P256       │ 64 字节   │ 64 字节           │
│ RSA-2048         │ 256 字节  │ 256 字节          │
│ ML-DSA-44        │ 2,420 字节│ 1,312 字节        │
│ ML-DSA-65        │ 3,293 字节│ 1,952 字节        │
│ ML-DSA-87        │ 4,595 字节│ 2,592 字节        │
└─────────────────┴──────────┴──────────────────┘
```

ML-DSA-44 的签名是 ECDSA-P256 的 **37 倍**，公钥是 **20 倍**。

### 这意味着什么？

在当前的 Web PKI 中，一次 TLS 握手需要传输：

- 服务器证书的签名和公钥
- 中间 CA 证书的签名和公钥
- 可能的 Certificate Transparency SCT 签名

如果直接替换为 ML-DSA，握手数据量会从约 4KB 膨胀到 **15-20KB**。对于一个每天处理数十亿请求的 CDN 来说，这不是一个可以忽略的开销。

更糟糕的是，**每个用户、每次连接**都要承受这个额外成本。在网络带宽有限的地区（移动端、发展中国家），这会显著影响用户体验。

### 为什么不直接用混合方案？

混合方案（hybrid key exchange，如 X25519MLKEM768）确实在部署中——Google 和 Cloudflare 已经在 TLS 密钥交换中启用了后量子算法。但混合方案解决的是**加密**问题，不是**认证**问题。

证书签名的尺寸问题需要不同的解决方案。这就是 Merkle Tree Certificates 的用武之地。

## Merkle Tree Certificates：用数据结构替代密码学

MTC 的核心思想可以用一句话概括：**不逐个签发证书，而是批量签发，用 Merkle 树的包含证明（inclusion proof）替代单个证书签名**。

### 传统证书签发流程

```
CA 签发证书：
  1. 收到域名验证请求
  2. 生成证书（含公钥）
  3. 用 CA 私钥签名
  4. 返回签名证书

TLS 握手：
  客户端收到 → 证书 + CA签名 + 中间CA证书 + CA签名 + SCT签名
  验证：3-4 次签名验证
```

### MTC 证书签发流程

```
CA 批量签发（MTC）：
  1. 收集一批证书请求（如每小时一批）
  2. 构建 Merkle 树，叶子节点是各证书的哈希
  3. 用 CA 私钥对 Merkle 根签名（一次签名覆盖整批）
  4. 发布"里程碑"（landmark）：签名 + Merkle 根
  5. 为每个证书生成 Merkle 包含证明

TLS 握手（MTC）：
  客户端收到 → 证书 + 1个签名 + 1个公钥 + 1个包含证明
  验证：1 次签名验证 + Merkle 路径验证
```

### 为什么 MTC 的握手更小？

关键在于：浏览器预先获取了里程碑（landmark），不需要在每次握手中传输 CA 签名。握手路径中只需要：

1. 服务器证书本身的签名和公钥（可以是后量子的）
2. 一个 Merkle 包含证明（哈希路径，通常几百字节）

文章原文说得非常明确：

> In the common case, the entire authentication path in an MTC handshake is one signature, one public key, and one inclusion proof. That's smaller than today's Web PKI handshake, even though MTCs use post-quantum algorithms.

即使使用 ML-DSA 这样的后量子算法，MTC 的握手数据量**比今天的 Web PKI 更小**。这是一个反直觉但极其优雅的工程优化。

## Merkle 树：一个被低估的密码学原语

Merkle 树（Merkle Tree）并不是新技术。它由 Ralph Merkle 在 1979 年提出，是区块链、Git、Certificate Transparency 等系统的核心数据结构。

```
         Root (已签名)
        /      \
      H(AB)    H(CD)
      /  \     /  \
    H(A) H(B) H(C) H(D)
     |    |    |    |
    证书A 证书B 证书C 证书D
```

**包含证明**的原理：要证明"证书 B 在树中"，只需要提供从 B 到 Root 的路径上的兄弟哈希值：

```
证明 B 的存在：H(A), H(CD)
验证：H(H(A) || H(B)) == H(AB)? ✓
     H(H(AB) || H(CD)) == Root? ✓
```

路径长度 = log₂(n)，其中 n 是批次中的证书数量。即使有百万级证书，路径也只需约 20 个哈希值。

### MTC 与 Certificate Transparency 的关系

Let's Encrypt 自 2019 年起就运营着自己的 CT（Certificate Transparency）日志。CT 日志本身就是 Merkle 树——这是追加写入的、不可篡改的数据结构。

MTC 可以看作是 CT 的自然演进：

| 特性 | 当前 CT | MTC |
|------|--------|-----|
| 证书签发 | 独立签名 | 批量签名 |
| 透明性 | 事后记录 | 内建于签发过程 |
| 签名验证 | 逐个验证 | 批量验证 |
| 撤销机制 | CRL/OCSP | Merkle 树更新 |

CT 是"先签发，后记录"，MTC 是"签发即记录"。这个架构简化了透明性保证，减少了攻击面。

## Go 1.27 的 ML-DSA 支持：基础设施开始就位

Let's Encrypt 的证书签发基础设施大量使用 Go 语言。Go 1.27 将内置 ML-DSA 支持，这意味着：

```go
// Go 1.27 预期 API（示意）
import "crypto/mldsa"

// 生成 ML-DSA-44 密钥对
privKey, err := mldsa.GenerateKey(rand.Reader, mldsa.Mode44)

// 签名
sig, err := privKey.Sign(rand.Reader, message, opts)

// 验证
err = privKey.Public().Verify(message, sig, opts)
```

这消除了一个关键的工程障碍：不再需要依赖 CGO 绑定或第三方密码学库来实现后量子签名。

每个环节都需要就位，整个迁移才能工作。这也是为什么 Let's Encrypt 说 "this is not a small endeavor"。

## 实际影响：开发者现在该做什么？

### 短期（2026-2027）

1. **启用混合后量子密钥交换**：如果你的服务器支持 X25519MLKEM768，现在就可以启用。这保护密钥交换免受"先存储后解密"攻击。

2. **检查你的 TLS 库版本**：OpenSSL 3.5+、Go 1.27+、BoringSSL 都已支持或即将支持 ML-DSA。

3. **关注 ACME 客户端更新**：如果你使用 certbot 或 acme.sh，关注它们对 MTC 的支持进度。

### 中期（2027-2029）

4. **更新证书处理代码**：如果你的应用程序解析或验证 X.509 证书，确保它能处理更大的签名和公钥。

5. **测试 MTC 兼容性**：当 Let's Encrypt 的 MTC 测试环境上线后，用你的应用测试证书申请和验证流程。

### 长期（2029+）

6. **审计信任链依赖**：检查你的系统是否依赖硬编码的 RSA/ECDSA 公钥或证书钉扎（certificate pinning）。这些需要迁移到后量子算法。

## 性能对比：MTC vs 传统 Web PKI

基于 Let's Encrypt 文章的描述，我们可以估算 MTC 的性能优势：

```
TLS 1.3 握手数据量估算：

传统 Web PKI (ECDSA-P256)：
  证书链签名：64 + 64 = 128 字节
  公钥：64 + 64 = 128 字节
  SCT 签名：64 字节 × 2 = 128 字节
  总计：~384 字节签名数据

直接替换为 ML-DSA-44：
  证书链签名：2420 + 2420 = 4840 字节
  公钥：1312 + 1312 = 2624 字节
  SCT 签名：2420 × 2 = 4840 字节
  总计：~12,304 字节签名数据（32倍增长！）

MTC + ML-DSA-44：
  服务器证书签名：2420 字节
  服务器公钥：1312 字节
  Merkle 包含证明：~640 字节（20层 × 32字节/哈希）
  总计：~4,372 字节签名数据

MTC 优势：比直接替换小 64%，比传统方案大 11 倍
```

关键洞察：MTC 不能让后量子签名"免费"，但它把**二次方增长**（证书链中每个证书都变大）变成了**线性增长**（只有终端证书变大）。

## 开放问题与社区讨论

HN 评论中提出了几个值得思考的问题：

**批次延迟问题**（LoganDark）：

> 如果批次执行太频繁，客户端数据库会太大；如果太不频繁，新证书的首次签发需要等待。

这是一个实际的工程权衡。Let's Encrypt 的"standalone"形式（更大的证书但不需要等待批次）可能是解决方案。

**后量子算法的可信度**（z3ratul163071）：

> NSA 和 EU 推动替换可靠算法为未经验证的后量子算法，这很可疑。

这是一个合理的担忧。ML-DSA 基于格问题（Learning With Errors），虽然经过多年密码学分析，但确实没有 RSA 和 ECC 那样的"实战检验"历史。这也是为什么混合方案（传统 + 后量子）是当前推荐的做法。

**量子计算机的时间线**（alansaber）：

> 量子未来的时间线是什么？20年？50年？100年？

没有人确切知道。但密码学迁移的工程周期是 5-10 年，而量子计算机的突破可能是突然的。**提前迁移是风险管理，不是过度反应**。

## 总结

Let's Encrypt 的 MTC 计划标志着后量子密码学从学术讨论进入工程实施阶段。核心要点：

1. **量子威胁不只是加密问题**：认证（证书签名）同样脆弱，长寿命密钥风险最高
2. **后量子签名尺寸爆炸**：ML-DSA 签名是 ECDSA 的 37 倍，直接替换会让 TLS 握手膨胀 32 倍
3. **MTC 是优雅的工程方案**：用 Merkle 树的批量签名 + 包含证明，把二次方增长变成线性增长
4. **生态系统正在就位**：Go 1.27 内置 ML-DSA，Chrome/Cloudflare 正在实验 MTC
5. **现在就应该行动**：启用混合密钥交换，关注 ACME 客户端更新，审计信任链依赖

后量子迁移是一场没有硝烟的基础设施战争。感谢 Let's Encrypt、Cloudflare、Google 和 Chrome 团队在前线推进。对于大多数开发者来说，最重要的一步是：**确保你的 TLS 配置支持混合后量子密钥交换**。这不是"以后再说"的事情——harvest now, decrypt later 攻击已经在发生了。

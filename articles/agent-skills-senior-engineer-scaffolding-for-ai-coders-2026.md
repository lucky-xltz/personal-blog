# Agent Skills：给 AI 编码代理装上「高级工程师骨架」

> 高级工程师的工作，大部分是 diff 里看不到的那些部分——写规格说明、写测试、做代码审查、控制变更范围、拒绝合并不确定的代码。AI 编码代理默认会跳过所有这些。Agent Skills 是我的一次尝试：让这些不再是可选项。

这是 Google Chrome 工程师 Addy Osmani 在 2026 年 5 月 3 日发布的博文开篇。他的开源项目 [Agent Skills](https://github.com/addyosmani/agent-skills) 已经获得 26K+ Star，成为 AI 编码代理领域最重要的工程实践框架之一。

今天这篇文章，我们不聊 Bun 和 Zig（上午已经聊过了），来聊聊一个更本质的问题：**当 AI 能写代码了，我们怎么确保它写的是「好」代码？**

---

## 问题：代理的「最短路径」陷阱

任何 AI 编码代理的默认行为都是走最短路径到达「完成」。你让它写一个功能，它就写一个功能。它不会问你有没有需求文档，不会先写测试再写实现，不会考虑这个变更是否跨越了信任边界，也不会想想这个 PR 对审查者来说是什么样子。它产出代码，宣布胜利，然后继续前进。

这和初级工程师的毛病一模一样。高级工程师花了整个职业生涯学习避免的事情——**表面假设、写规格说明、把工作拆成可审查的块、选择无聊但可靠的方案、留下正确性的证据、把变更控制在人类能审查的范围内**——代理统统跳过。

为什么？因为这些步骤是「不可见的」。奖励信号指向的是「任务完成」，而不是「任务完成且设计文档存在」。

```
初级工程师 / AI代理的工作模式：
┌─────────────┐
│  收到需求     │
│     ↓        │
│  写代码       │  ← 最短路径
│     ↓        │
│  宣布完成 ✓   │
└─────────────┘

高级工程师的工作模式：
┌─────────────┐
│  收到需求     │
│     ↓        │
│  表面假设     │  ← 不在 diff 里
│     ↓        │
│  写规格说明   │  ← 不在 diff 里
│     ↓        │
│  拆分工作     │  ← 不在 diff 里
│     ↓        │
│  选择无聊方案  │  ← 不在 diff 里
│     ↓        │
│  写测试       │
│     ↓        │
│  写代码       │
│     ↓        │
│  留下证据     │  ← 不在 diff 里
│     ↓        │
│  控制范围     │  ← 不在 diff 里
│     ↓        │
│  提交审查 ✓   │
└─────────────┘
```

Agent Skills 的核心理念就是：**把高级工程师的骨架重新装回去**。

## 什么是「Skill」？

在 Claude Code / Anthropic 的语境里，「Skill」这个词承载了太多含义。精确地说：

**Skill 是一个带 frontmatter 的 Markdown 文件，在合适的场景下被注入到代理的上下文中。** 它介于系统提示片段和运维手册之间。

关键区别：Skill **不是**参考文档。它不是「关于测试你应该知道的一切」。它是一个**工作流**——代理按顺序执行的步骤序列，每个步骤都有检查点产出证据，最终到达一个定义好的退出条件。

```markdown
# 一个 Skill 的结构示例
---
name: test-driven-development
description: 测试驱动开发工作流
triggers:
  - "实现新功能"
  - "修复 bug"
---

## 步骤 1：写失败的测试
- 根据需求编写最小的失败测试
- 运行测试，确认它失败了
- ✅ 退出条件：测试运行且输出红色

## 步骤 2：写最小实现
- 写刚好能让测试通过的代码
- 不要多写一行
- ✅ 退出条件：测试变绿

## 步骤 3：重构
- 在测试保护下重构
- 确认测试仍然通过
- ✅ 退出条件：测试绿 + 代码更简洁
```

这个区别是整个项目的核心。如果你把一篇 2000 字的测试最佳实践文章放进代理的上下文，代理会读它，生成看起来合理的文字，然后跳过实际的测试。如果你放的是一个**工作流**（先写失败的测试、运行它、看它失败、写最小代码让它通过、看它通过、重构），代理就有事可做，你就有东西可以验证。

**流程优于散文。工作流优于参考。有退出条件的步骤优于没有退出条件的长文。**

## Skill 编码的 SDLC

Agent Skills 仓库中的 20 个 Skill 围绕六个生命周期阶段组织，上面挂着 7 个斜杠命令：

| 阶段 | 命令 | 做什么 |
|------|------|--------|
| Define | `/spec` | 决定你要构建什么 |
| Plan | `/plan` | 拆分工作 |
| Build | `/build` | 垂直切片实现 |
| Verify | `/test` | 证明它能工作 |
| Review | `/review` | 捕捉遗漏的问题 |
| Ship | `/ship` | 安全地交付给用户 |
| Simplify | `/code-simplify` | 贯穿全流程的简化 |

这和每个正常运转的工程团队的 SDLC 是一样的，只是换了个说法：

- **Google 叫它**：设计文档 → 评审 → 实现 → 可读性审查 → 发布检查清单
- **Amazon 叫它**：逆向工作备忘录 + 提高标准的人
- **每个健康团队都有**：这个循环的某个版本

新的地方在于：**大多数 AI 编码代理默认跳过了这些阶段的大部分。** 你要求一个功能，你得到一个实现，而规格说明、计划、测试、审查和发布清单统统不会发生。

一个复杂的功能可能按顺序激活 11 个 Skill。一个小 bug 修复可能用 3 个。路由器（`using-agent-skills`）决定哪个适用。工作流的规模跟着实际范围走，而不是跟着假定范围走。

## 五个承重设计决策

Agent Skills 项目中有五个设计决策是承重的，其余都从它们派生。

### 1. 流程优于散文（Process over Prose）

工作流是代理可执行的；散文不是。人类团队也一样——如果你的团队手册有 200 页，没人在时间压力下读它。如果它是一小组带检查点的工作流，人们真的会执行。

### 2. 反合理化表格（Anti-Rationalization Tables）

这是项目中最有特色的设计决策，也是 Addy 最希望其他团队「偷走」的。

每个 Skill 都包含一个表格，列出代理（或疲惫的工程师）可能用来跳过工作流的常见借口，配上预先写好的反驳：

| 借口 | 反驳 |
|------|------|
| 「这个任务太简单，不需要规格说明」 | 验收标准仍然适用。五行可以，零行不行 |
| 「我之后再写测试」 | 「之后」是承重词。没有之后。先写失败的测试 |
| 「测试通过了，发布吧」 | 测试通过是证据，不是证明。你检查了运行时吗？验证了用户可见的行为吗？人类读了 diff 吗？ |

为什么这有效？因为 **LLM 极其擅长合理化**。它能生成听起来很有道理的段落，解释为什么这个特定任务不需要规格说明，或者为什么这个特定变更可以不经审查就合并。反合理化表格是预先写好的反驳——反驳代理还没说出口的谎话。

这个模式对人类团队同样有效。大多数工程衰退不是有人选择做坏工作，而是人们接受听起来合理的借口，跳过他们不想做的部分。一个写下自己反合理化表格的团队，就是一个有更少合理化的团队。

### 3. 验证是不可谈判的（Verification is Non-Negotiable）

每个 Skill 都以具体证据终止。测试通过。构建输出干净。运行时 trace 显示预期行为。审查者签字。「看起来对」永远不够。

这和 Anthropic 的 harness 从失败中恢复的原则一样，和 Cursor 的 planner/worker/judge 分裂实际捕获 bug 的原则一样，和任何长时间运行的代理可恢复的原则一样。**代理是生成器。你需要一个独立的信号来确认工作完成。** Skill 把这个信号烘焙进了每个工作流。

### 4. 渐进式披露（Progressive Disclosure）

不要在会话开始时加载全部 20 个 Skill。根据阶段激活它们。一个小型元 Skill（`using-agent-skills`）充当路由器，决定当前任务适用哪个 Skill。

这是 harness 工程在 Skill 粒度上的应用。每个加载到上下文的 token 都会在某处降低性能，所以你只加载相关的，把其余的留在磁盘上。渐进式披露是如何把一个 20 个 Skill 的库塞进 5K token 的槽位而不污染水源的。

### 5. 范围纪律（Scope Discipline）

元 Skill 编码了一条如果可以的话会钉在每个代理上的不可谈判规则：**只碰你被要求碰的东西。** 不要重构相邻系统。不要删除你不完全理解的代码。不要碰到一个 TODO 就决定重写文件。

这听起来很明显，直到你看到一个代理决定修复一个 bug 需要现代化三个不相关的文件。**范围纪律是代理的 PR 能否被合并还是必须回滚的最大决定因素。** 它也最清晰地映射到 Google 的代码审查规范——审查者会因为 PR 做了不止一件事而阻止它。

## 实战：如何构建自己的 Skill 系统

你不需要直接用 Agent Skills 仓库。理解原理后，你可以为自己的团队和工具链构建定制化的 Skill 系统。

### 方式一：直接使用（推荐大多数人）

如果你用 Claude Code：

```bash
# 安装 Agent Skills
/install-agent-skills
```

你得到斜杠命令（`/spec`、`/plan`、`/build`、`/test`、`/review`、`/ship`、`/code-simplify`），代理会根据上下文自动激活相关 Skill。

### 方式二：移植到你的工具

Skill 是纯 Markdown 加 frontmatter。你可以把它们放到任何支持系统提示的工具中：

```bash
# Cursor
cp skills/*.md .cursor/rules/

# Gemini CLI
cp skills/*.md .gemini/rules/

# 通用方案：放入 AGENTS.md
cat skills/test-driven-development.md >> AGENTS.md
```

### 方式三：从零构建（最有价值）

为你的团队定制 Skill。以下是一个最小可行的 Skill 模板：

```markdown
---
name: api-change-workflow
description: API 变更工作流
triggers:
  - "修改 API 接口"
  - "新增 API 端点"
anti_rationalizations:
  - excuse: "这个 API 内部使用，不需要文档"
    rebuttal: "内部 API 也有消费者。三个月后你自己就是那个消费者。"
  - excuse: "改动很小，不需要版本号"
    rebuttal: "Hyrum'''s Law：你 API 的每个可观察行为最终都会被人依赖。"
---

## 步骤 1：列出影响范围
- 搜索所有调用该 API 的代码
- 确认变更的向后兼容性
- ✅ 退出条件：影响范围清单完成

## 步骤 2：写 API 变更文档
- 变更前后的接口定义
- 迁移路径（如果有破坏性变更）
- ✅ 退出条件：文档存在且完整

## 步骤 3：实现 + 测试
- 先写失败的集成测试
- 实现变更
- 确认所有测试通过
- ✅ 退出条件：全绿

## 步骤 4：审查清单
- [ ] 向后兼容？是/否，理由
- [ ] 文档更新？是/否
- [ ] 相关测试更新？是/否
- [ ] 变更范围控制在 PR 描述内？是/否
```

## 五个「偷了就能用」的模式

即使你永远不安装任何 Skill，以下五个模式也值得偷走：

### 1. 反合理化作为团队实践

写下你的团队对自己说的谎。「我们发布后再修测试」「这个变更太小不需要设计文档」「没事，我们有监控」。给每个配上反驳。放进你的 AGENTS.md 或工程 Wiki。它会节省争论，也会捕捉下一个疲惫的周五下午的捷径。

### 2. 内部文档用流程代替散文

如果你发现自己在写一篇标题为「我们如何处理 X」的 2000 字文档，你写的是参考材料。把它转换成带检查点的工作流。文档缩减到 400 字，人们真的会执行。这适用于入职指南、运维手册，和任何人在时间压力下会读的东西。

### 3. 验证作为硬退出条件

让「产出证据」成为每个任务的退出步骤。对代理、对工程师、对你自己。证据是证明工作完成的任何东西：绿色的测试运行、截图、日志、审查批准。没有它，任务就没有完成。

### 4. 渐进式披露用于任何规则手册

不要写 50 页的手册。写一个小路由器，在合适的场景指向正确的小章节。这对 AGENTS.md、运维手册、事件手册，和任何人在时间压力下会读的东西都适用。

### 5. 五个不可谈判

从元 Skill 中提炼的五条规则，值得放进任何 AGENTS.md：

1. **构建前表面假设**——沉默的错误假设是最常见的失败模式
2. **需求冲突时停下来问**——不要猜
3. **有理由时推回去**——代理（或工程师）不是应声虫
4. **偏好无聊、显而易见的方案**——聪明是昂贵的
5. **只碰你被要求碰的东西**

## Google DNA：被编码的工程文化

Agent Skills 中渗透了 Google 的工程实践。这不是巧合——大部分让 Google 规模软件工作的东西都是公开文档化的，而这些恰恰是代理最可能跳过的。

| 实践 | 来源 | 在哪个 Skill 中 |
|------|------|----------------|
| Hyrum'''s Law | Software Engineering at Google | api-and-interface-design |
| 测试金字塔 (~80/15/5) | Google 测试博客 | test-driven-development |
| Beyoncé Rule | Google 测试博客 | test-driven-development |
| DAMP > DRY（在测试中） | Google 测试哲学 | test-driven-development |
| ~100 行 PR 限制 | Google 代码审查规范 | code-review-and-quality |
| Chesterton'''s Fence | 工程常识 | code-simplification |
| Trunk-based 开发 | Google 工程实践 | git-workflow-and-versioning |
| Shift Left + Feature Flags | DevOps 最佳实践 | ci-cd-and-automation |
| Code as Liability | 长期维护哲学 | deprecation-and-migration |

这些都不是新想法。**重点是它们默认不在代理里。** 一个前沿模型在训练数据中读到过「Hyrum'''s Law」这个短语，但它在凌晨 3 点设计你的 API 时不会应用 Hyrum'''s Law。Skill 是你确保它会的方式。

## 在 Agent Harness 中的位置

在更广阔的图景中，Skill 是代理 harness 工程的一层。Harness 是模型加上你围绕它构建的所有东西：

```
┌─────────────────────────────────────────┐
│            Agent Harness                │
│                                         │
│  ┌─────────┐  ┌──────────┐  ┌────────┐ │
│  │ Skills  │  │ AGENTS.md│  │ Hooks  │ │
│  │ 工作流   │  │ 规则手册  │  │ 确定性 │ │
│  │         │  │          │  │ 执行层  │ │
│  └─────────┘  └──────────┘  └────────┘ │
│                                         │
│  ┌─────────┐  ┌──────────────────────┐ │
│  │ Tools   │  │    Session Log       │ │
│  │ 可执行的 │  │    持久记忆          │ │
│  │ 动作    │  │                      │ │
│  └─────────┘  └──────────────────────┘ │
│                                         │
│  ┌─────────────────────────────────┐   │
│  │           Model                 │   │
│  └─────────────────────────────────┘   │
└─────────────────────────────────────────┘
```

- **Skills** 做高级工程师流程的工作
- **AGENTS.md** 做滚动规则手册的工作
- **Hooks** 做确定性执行层的工作（比如 git pre-commit hook）
- **Tools** 做代理可以采取的动作
- **Session Log** 做持久记忆的工作

Skill 对长时间运行的代理比对聊天式代理更重要，因为长时间运行会放大每个捷径。一个在 10 分钟会话中跳过测试的代理产生一个 bug。一个在 30 小时会话中跳过测试的代理，在运行结束时产生一个考古学级别的调试项目——那时候没人记得原始意图是什么了。

## 总结

AI 编码代理是有史以来最有能力的初级工程师，但对工作中不体现在 diff 里的部分没有本能。高级工程师的工作——表面假设、控制变更大小、写规格说明、留下证据、拒绝合并不确定的代码——除非你让它不可能被跳过，否则代理一定会跳过。

Agent Skills 给出的答案很简单：**编码纪律，让代理无法自我说服跳过。**

不管你是安装现成的 Skill、移植到自己的工具链，还是从零构建，核心教训都一样：**高级工程师的部分不再是可选项，即使工程师是一个模型。**

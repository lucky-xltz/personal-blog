# 当编译器开始思考：Rust 项目如何为 LLM 立规矩

2026 年 5 月，Rust 编译器项目（`rust-lang/rust`）正式提交了一份 [LLM 使用政策 PR](https://github.com/rust-lang/rust-forge/pull/1040)，在 Hacker News 上引发了 80+ points、44 条评论的热烈讨论。这不是又一个"AI 编码规范"——它是目前主流开源项目中，对 LLM 参与贡献最细致、最有思考深度的治理框架。

核心原则只有一句话：**LLM 可以用来思考，但不能用来创造。**

## 为什么需要这份政策？

Rust 项目面临一个独特的困境：

1. **社区分裂**：一部分成员认为 AI 是强大工具，另一部分认为其社会和环境代价不可接受，还有人在观望
2. **质量底线**：Rust 以代码质量著称，"vibes-coded" 的 PR 会侵蚀这个声誉
3. **信任基础**：开源贡献建立在互信之上，LLM 生成的代码模糊了"谁真正理解这段代码"的边界

这份政策的设计目标不是禁止 LLM，而是在"技术质量"和"社区信任"之间找到平衡点。

## 政策的核心框架：三色分级

政策用四个等级来分类不同的 LLM 使用场景：

### ✅ 允许：私有使用，不产生公开输出

```text
- 用 LLM 询问现有代码库的问题
- 用 LLM 总结 issue/PR/RFC 的评论（仅限个人使用，不得公开转发）
- 用 LLM 私下 review 自己的代码
- 用 LLM 生成解决方案的灵感，然后从零开始用自己的风格重写
```

关键洞察：**LLM 作为"思考伙伴"是完全允许的**。你可以用它理解代码、分析问题、寻找灵感——只要最终的输出是你自己写的。

### ❌ 禁止：用 LLM 代替人类创造

```text
- 用 LLM 原创的评论（issue body、PR 描述也算）
- 用 LLM 原创的文档（doc-comment、safety comment、编译器诊断消息）
- 把 LLM review 当作合并/拒绝的充分条件
```

为什么禁止编译器诊断消息用 LLM 生成？因为诊断消息是 Rust 与开发者沟通的核心界面。LLM 可以辅助诊断的**逻辑**，但消息的文字必须是人类精心打磨的。

### ⚠️ 需披露：灰色地带

```text
- 机器翻译（必须同时附上原文）
- 不达到"独创性"门槛的琐碎修改
- 用 LLM 发现 bug（但必须自己验证和撰写报告）
- LLM review bot（必须用独立 GitHub 账号，必须可被用户屏蔽）
```

Review bot 的规则尤其值得注意：

```text
1. 必须有独立的 GitHub 账号，标识为 LLM
2. 不能未经维护者批准就自动发评论
3. 不能用 GitHub "App" 账号（因为用户无法屏蔽）
4. LLM 的评论不能是 blocking 的——reviewer 必须明确背书才能阻塞 PR
5. 如果有更可靠的工具（linter、formatter），应该优先使用
```

## 最有趣的实验：LLM 原创代码的例外

政策中最引人注目的部分是一个"实验性例外"——在严格条件下，允许 LLM 原创的代码变更：

```text
1. 被邀请的（Solicited）：reviewer 提前同意审查 LLM 原创的 PR
   - 新贡献者必须先与 reviewer 沟通，不能直接提交
2. 非关键的（Non-critical）：不太可能导致 soundness 回归
   - 内部工具如 tidy、x setup、linkchecker → 可以
   - trait 系统、MIR 构建、query 系统 → 不行
3. 高质量的（High-quality）：至少与其他代码变更同一标准
4. 充分测试的（Well-tested）：覆盖所有能想到的边界情况
   - 没有"写测试太难"的例外
   - LLM 原创 PR 的测试标准比人类更高，因为 LLM 更容易写测试
5. 充分 review 的（Well-reviewed）：作者和 reviewer 都必须完全理解代码
   - 推荐用第二个 LLM 做对抗性 review
```

所有 LLM 原创 PR 都会打上 `ai-assisted` 标签，并转发到一个私有 Zulip 频道。这个频道的目的不是额外审查，而是**收集实验数据**：LLM 是否在帮助人们做有意义的事？人们是否在学习？是否在持续贡献？

## 政策背后的哲学

### "原作者"的定义

政策对"originally authored"的定义非常精准：

> "由 LLM 生成的文本（可能经过人类编辑）"。任何程度的编辑都不能改变作者身份——作者身份决定了初始风格，而风格一旦确定就很难改变。

这引用了经典的 [What Colour are your bits?](https://ansuz.sooke.bc.ca/entry/23) 文章——关于"来源"如何影响代码的"颜色"（性质）。

### "最优欺诈量不是零"

政策的版主指南中引用了一篇金融文章的洞察：

> 不要试图当警察。如果明确违反了规则，指向这份政策；如果在灰色地带，报告给版主然后继续你的工作。

这是一种务实的态度——完全禁止 LLM 使用既不现实也不必要，关键是建立清晰的边界和披露机制。

### 责任归属

> 你的贡献是你的责任；你不能把责任推给 LLM。

这一条简洁有力。无论你用了什么工具，最终对代码负责的是你——这个原则在 AI 时代比以往任何时候都重要。

## 对其他项目的启示

Rust 的这份政策为开源社区提供了一个可参考的治理模板：

| 原则 | 实践 |
|------|------|
| 区分"思考"和"创造" | 私有使用完全放开，公开输出严格管控 |
| 分级而非一刀切 | 用颜色标记不同场景，留有灰色地带 |
| 披露而非禁止 | 要求声明 LLM 使用，而非完全禁止 |
| 实验性框架 | 为 LLM 原创代码设置试验场，收集数据再决策 |
| 责任明确 | 人类对所有贡献负最终责任 |

## 总结

Rust 项目的 LLM 政策不是一份恐 AI 的保守文件，也不是一份放任自流的开放宣言。它是一个精心设计的治理实验——在尊重社区分歧的同时，为 LLM 的合理使用划出清晰的边界。

核心信息是：**LLM 是工具，不是作者。** 你可以用它思考、分析、发现，但最终的文字和代码必须出自你的理解。这种"人机协作"而非"人机替代"的理念，或许是 AI 时代开源治理的最佳答案。

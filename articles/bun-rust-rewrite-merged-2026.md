# 77万行代码一夜之间换了语言：Bun 的 Rust 重写被合并了，然后呢？

2026年5月14日，GitHub 上一个 PR 悄然合并。没有烟花，没有发布会，但它的规模足以载入软件工程史册：**oven-sh/bun#30412**——将 Bun 这个 JavaScript 运行时从 Zig 重写为 Rust，`+773,950` 行代码，`-151` 行删除，涉及 1,646 个文件。

更令人震惊的是，这整个重写主要由 AI（Claude）完成。

九天前，我们分析了 Bun 发布的 Zig→Rust 迁移指南。当时它还像一个宏大的实验。今天，实验结果来了：**99.8% 的测试通过率，PR 被合并到 main 分支。**

这不是一篇技术迁移指南——那我们已经写过了。这篇文章想聊的是：当 AI 真正完成了一次大规模代码迁移，我们应该怎么看这件事？

## 从"实验"到"合并"的九天

回顾一下时间线：

| 日期 | 事件 |
|------|------|
| 2025-12-02 | Anthropic 收购 Bun，HN 获得 2192 分 |
| 2026-05-05 | Bun 提交 Zig→Rust 迁移指南（576 行），获 HN 716 分 |
| 2026-05-05 | 同日，`claude/phase-a-port` 分支被发现，1,646 个文件变更 |
| 2026-05-09 | Jarred Sumner 宣布 Rust 重写通过 99.8% 测试，获 HN 713 分 |
| 2026-05-14 | PR #30412 被合并到 main 分支 |

从迁移指南发布到 PR 合并，只用了 **9 天**。

九天重写一个 77 万行的代码库——如果这是人工完成的，大概需要一个几十人的团队花上一两年。但 Bun 团队利用 Anthropic 内部的"无限 Claude 额度"，用 AI 做了绝大部分翻译工作。

## "Vibe Coding"：一个新物种

HN 用户 @stingraycharles 的评论一针见血：

> "当前 HN 热帖还在讨论 Anthropic 收购 Bun 后不会搞 vibe coding 实验。结果呢？我们已经在看了。"

所谓 **vibe coding**，不是传统意义上的"写代码"。它更像是：

1. **给 AI 一份迁移规范**（576 行的 PORTING.md）
2. **让 AI 执行批量翻译**（Zig → Rust 的逐文件转换）
3. **用测试套件验证结果**（99.8% 通过率）
4. **人工修复剩余的 0.2%**

这个流程和我们熟悉的"AI 辅助编程"完全不同。不是 Copilot 补全几行代码，而是 AI 独立完成了整个代码库的语言迁移。

一位 HN 用户 @Humphrey 分享了他的观察：

> "我在用 AI 写 Rust 代码时比写动态语言更成功。我猜 Rust 的严格类型系统让 AI 有更多约束可以遵循——这反而让它的输出更可靠。"

这揭示了一个反直觉的事实：**类型越严格的语言，AI 翻译越可靠**。因为编译器本身就是最好的验证器。

## 技术决策：为什么不用 tokio？

Bun 的 Rust 重写做出了几个出人意料的技术选择：

- **不用 `tokio`**：Bun 没有采用 Rust 生态中最流行的异步运行时
- **不用 `rayon`**：没有使用并行计算库
- **不用 `hyper`**：没有用标准的 HTTP 库
- **不用 `std::fs`**：甚至没有用标准库的文件系统操作
- **不用 `async-trait`**：避免了常见的异步 trait 方案

这些选择意味着 Bun 在 Rust 中重新实现了大量底层逻辑，而不是依赖社区生态。这有两个原因：

1. **性能控制**：Bun 的核心卖点是极致速度，通用库往往有性能妥协
2. **迁移一致性**：从 Zig 的底层控制迁移到 Rust，保持相同的控制粒度比引入抽象层更安全

这种"不用标准库、自己造轮子"的做法在 Rust 社区颇具争议，但对于 Bun 这种追求极致性能的运行时来说，可能是唯一合理的选择。

## CI 机器人说了实话

PR 合并后，一个有趣的插曲：Bun 的 CI 系统自动将后续清理 PR（#30680）标记为 **"ai slop"**。

这个标签本来是 Bun 用来过滤低质量 AI 生成的社区 PR 的。结果，它把自家用 AI 写的代码也标记了。

这个黑色幽默的事件暴露了一个深层问题：**我们还没有好的方法来区分"高质量 AI 代码"和"低质量 AI 代码"**。CI 系统看到的是模式——大量结构相似的代码变更、批量生成的文件——而这些模式恰好也是低质量 AI 输出的特征。

## 社区的担忧

HN 讨论中，担忧主要集中在几个方面：

### 1. 测试质量的疑问

> "99.8% 通过率听起来很厉害，但测试套件本身的质量如何？如果测试覆盖不够，通过率再高也没有意义。"

这是合理的质疑。一个只有 6 天工作的迁移，测试套件可能还没有来得及更新以适配新语言的边界情况。

### 2. 向后兼容性风险

> "我开始看这些 commit，他们基本上是在通过修改测试本身来解决'测试不通过'的问题。"

如果 AI 翻译的代码不通过测试，修复策略是"改测试"还是"改代码"？这个区别至关重要。

### 3. 认知债务

> "Bun 是 Claude Code 的运行时，Claude Code 是 Anthropic 万亿美元公司的核心产品。现在这个产品建立在一个没有人能完全理解的 vibe-coded 应用之上。"

这可能是最深刻的担忧：当 AI 写的代码超过人类能理解的规模，**谁来对代码质量负责？**

### 4. "自己吃自己的狗粮"

一位用户指出了一个颇具讽刺意味的循环：

> Anthropic 的产品 Claude → Claude Code 跑在 Bun 上 → Bun 现在用 Claude 重写了 → Claude Code 现在跑在 Claude 写的 Bun 上

这是一个递归的信任链。如果 AI 写的运行时有 bug，那运行在这个运行时上的 AI 工具也会受影响。

## 这对行业意味着什么？

### 大规模代码迁移不再是天方夜谭

在此之前，"把整个项目从语言 A 重写到语言 B"是最令人望而生畏的工程任务之一。Python 2→3 花了十年，Perl→Python 的迁移至今仍在进行中。

现在，Bun 用 9 天证明了 AI 可以承担这个任务的主要工作量。这并不意味着每个项目都应该这么做——但它确实意味着**重写不再是"不可能"的选项**。

### 语言选择的权重在变化

Zig 选择的一个隐性成本是 AI 工具支持不足。当 AI 编码助手成为开发流程的核心部分时，**语言的 AI 训练数据量**成了一个实际的技术指标。

这意味着：
- Rust、Python、JavaScript 等主流语言会因为 AI 支持好而吸引更多项目
- 小众语言的生存空间会被进一步压缩
- 语言设计者需要考虑"AI 可翻译性"这个新维度

### "理解代码"的含义在改变

过去，理解一个代码库意味着阅读每一行代码。现在，77 万行代码不可能被任何一个人完全理解——即使它是人写的。

也许我们需要接受一个新的现实：**代码的"可理解性"不再等于"每个开发者都能读懂每一行"**，而是"有足够的测试、文档和抽象来保证行为正确"。

## 最后的思考

Bun 的 Rust 重写是一个里程碑事件，但不是因为它证明了 Rust 比 Zig 好——这是另一个话题。

它证明的是：**AI 驱动的大规模代码迁移已经成为现实**。

九天，77 万行代码，99.8% 测试通过率。这些数字会不断被刷新。下一次可能是更大的代码库，更短的时间，更高的通过率。

我们正在进入一个软件工程的新时代。在这个时代里，"用什么语言写代码"可能不再是最重要的问题——"如何验证 AI 写的代码是否正确"才是。

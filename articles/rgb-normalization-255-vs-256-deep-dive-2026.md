# RGB 归一化之争：除以 255 还是 256？

每个写过图像处理代码的人都见过这行代码：`pixel / 255.0`。它把 8 位整数映射到 `[0, 1]` 浮点范围，是 GPU、OpenCV、Pillow 等几乎所有图像处理框架的标准做法。

但最近一篇技术文章在 Hacker News 上引发了 90 多条激烈讨论：**我们应该除以 255 还是 256？** 这个看似微不足道的选择背后，藏着量化理论、浮点精度和信号处理的深层问题。

## 一、两种归一化方案

假设你正在编写一个图像处理程序：读取 8 位 RGB 图像，转换为浮点数处理，最后保存回 8 位整数。归一化步骤有两种写法：

**方案 A：标准方案（除以 255）**

```python
# 加载：整数 → 浮点
pixels = img / 255.0        # 0 → 0.0, 255 → 1.0

# 处理
result = process(pixels)

# 保存：浮点 → 整数
output = np.trunc(result * 255 + 0.5)
output_8bit = output.clip(0, 255).astype(np.uint8)
```

**方案 B：替代方案（除以 256）**

```python
# 加载：整数 → 浮点（加 0.5 偏置）
pixels = (img + 0.5) / 256.0   # 0 → 0.00195, 255 → 0.99805

# 处理
result = process(pixels)

# 保存：浮点 → 整数
output = np.trunc(result * 256)
output_8bit = output.clip(0, 255).astype(np.uint8)
```

两者的核心区别在于：

| 特性 | 方案 A（÷255） | 方案 B（÷256） |
|------|---------------|---------------|
| 0 映射到 | 0.0 | 0.00195 |
| 255 映射到 | 1.0 | 0.99805 |
| 区间宽度 | 255 份 | 256 份 |
| 零点行为 | Mid-riser | Mid-tread |
| 128 映射到 | 0.50196... | 精确的 0.5 |

方案 B 的支持者认为它"更整齐"——每个浮点值精确落在两个整数的正中间，128 映射到精确的 0.5 而非 0.50196...。但方案 A 的支持者有更强的论据。

## 二、可视化理解：数轴上的"箱子"

要理解两者的差异，最好的方式是在数轴上可视化。以 3 位整数（0-7）为例，每个整数对应一个"箱子"（bin），箱子的宽度代表该整数在浮点反量化时的覆盖范围。

```
方案 A (÷7):  数轴 [0, 1]
  0    1    2    3    4    5    6    7
  |----|----|----|----|----|----|----|
  ←宽→                              ←宽→
  箱子0 箱子1 箱子2 箱子3 箱子4 箱子5 箱子6 箱子7

方案 B (÷8):  数轴 [0, 1]
   0   1   2   3   4   5   6   7
  |----|----|----|----|----|----|----|----|
  均匀分布，每个箱子宽度相等
```

方案 A 的两端箱子（0 和 255）只有中间箱子的一半宽度——它们被"拉伸"到了 `[0, 1]` 范围之外。这意味着：

- **均匀噪声分布不均**：如果你生成 `[0, 1]` 均匀随机数并用方案 A 的公式量化，0 和 255 出现的概率只有其他值的一半
- **极端值更难输出**：纯黑（0）和纯白（255）在浮点处理中需要更大的偏移才能被"选中"

用代码验证：

```python
import numpy as np
import matplotlib.pyplot as plt

# 生成 100 万个均匀随机数
result = np.random.uniform(0, 1, 1_000_000)

# 方案 A 量化
final_a = np.trunc(result * 255 + 0.5).clip(0, 255).astype(np.uint8)

# 方案 B 量化
final_b = np.trunc(result * 256).clip(0, 255).astype(np.uint8)

fig, (ax1, ax2) = plt.subplots(1, 2, figsize=(12, 4))
ax1.hist(final_a, bins=256, range=(0, 255), alpha=0.7)
ax1.set_title('方案 A: ÷255 — 注意两端的半高柱')
ax2.hist(final_b, bins=256, range=(0, 255), alpha=0.7)
ax2.set_title('方案 B: ÷256 — 均匀分布')
plt.tight_layout()
plt.show()
```

运行这段代码，你会清楚地看到方案 A 的直方图两端有两个"矮柱"。

## 三、量化理论：Mid-riser vs Mid-tread

从信号处理的角度看，这两种方案对应两种经典的均匀量化器：

**Mid-riser 量化器**（方案 A）：
- 零点位于台阶的"竖板"上（不对应任何整数输出）
- 没有精确的零输出
- 适用于无符号数据（如 RGB 像素）

**Mid-tread 量化器**（方案 B）：
- 零点位于台阶的"踏板"上（对应整数 0）
- 有精确的零输出
- 适用于有符号数据（如音频信号）

这个术语来自一篇 1977 年的经典论文。当你把量化器的输入-输出函数画成阶梯形状时，"mid-riser"在零点处踩在台阶的竖板上（rising edge），而"mid-tread"踩在踏板上（tread）。

量化步长公式：

```
Mid-riser: y_k = (k + 0.5) × Δ    （Δ = 2/L，L 为量化级数）
Mid-tread: y_k = k × Δ
```

标准公式 `÷255` 实际上是一个 **L=255 的 Mid-riser 量化器**——这在数学上有些奇怪，因为 Mid-riser 通常用于有符号输入，而 L=255 也不是 2 的幂。方案 B `÷256` 则是一个 **L=256 的 Mid-tread 量化器**，更符合教科书定义。

## 四、但你应该用哪个？

理论分析似乎偏向方案 B，但实际工程中，**方案 A（除以 255）几乎总是正确选择**。原因如下：

### 4.1 你处理的是别人的图像

大多数 RGB 图像在保存时使用的是标准公式（÷255）。如果你用方案 B（÷256）加载这些图像，会引入额外误差：

```python
# 假设原始像素值 128 是用 ÷255 量化的
# 正确加载：128 / 255 ≈ 0.50196
# 错误加载：(128 + 0.5) / 256 = 0.50195

# 误差很小（~0.00002），但累积起来会影响处理结果
```

在实际应用中，颜色值不是绝对物理测量——即使 sRGB 规范声称如此——这点误差通常可以忽略。但**混用编码和解码方案是明确的 bug**，这在大型代码库中是很容易犯的错误。

### 4.2 GPU 和硬件使用方案 A

现代 GPU 在纹理采样和帧缓冲操作时，硬件层面使用的就是 `÷255` 归一化。如果你的软件处理与 GPU 管线不一致，会导致微妙的颜色偏差。一致性比理论优雅更重要。

### 4.3 零值映射很重要

方案 B 将整数 0 映射到 0.00195 而非精确的 0.0。这意味着你的图像处理代码无法用 `== 0.0` 检测纯黑像素。虽然可以用阈值比较，但这增加了不必要的复杂性。

### 4.4 极端值偏差在实践中无害

方案 A 的两端箱子只有正常宽度的一半，但这真的重要吗？Hacker News 评论者 `herf` 指出了一个有趣的角度：

> RGB 值代表的是相对于某个适应状态的亮度，而日光场景中的"零"并不是"零亮度"——它大约只有最亮点的 0.001 倍——仍然是数百万个光子，远非真正的零。从某种意义上说，我们的眼睛在滑动标尺上体验对比度，没有真正的"零点"。

换句话说，物理世界中的"纯黑"几乎不存在，方案 A 在极端值上的理论偏差在实际图像处理中几乎无法察觉。

## 五、音频世界的类似问题

同样的问题在音频领域以略有不同的形式存在。16 位整数音频的范围是 `[-32768, 32767]`——注意它是**不对称的**！而浮点音频使用 `[-1.0, 1.0]`。

```python
# 16位音频 → 浮点
float_sample = int_sample / 32768.0   # 注意：不是 32767

# 浮点 → 16位音频
int_sample = int(float_sample * 32768)
```

这里使用 32768 而非 32767，是因为有符号整数的特殊性质：`-32768` 没有对应的正数（`32768` 溢出了）。这和 RGB 的 255 vs 256 问题本质上是同一个数学问题：**如何在整数和浮点之间建立最优映射**。

## 六、一个精妙的混合方案

HN 评论者 `atilimcetin` 提出了一个混合方案，同时获得两个世界的好处：

```python
# 浮点 → 整数：用 ÷256 的均匀分布
i = min(int(f * 256), 255)

# 整数 → 浮点：用 ÷255 的精确映射
f = i / 255.0
```

这个方案的关键特性：**对所有 256 个整数值，`uint8 → float → uint8` 的往返转换完全无损**。

验证：

```python
for i in range(256):
    f = i / 255.0
    recovered = min(int(f * 256), 255)
    assert recovered == i, f"Roundtrip failed for {i}: got {recovered}"
print("所有 256 个值往返转换成功！")
```

为什么有效？因为 `÷255` 在编码时将每个整数映射到其箱子的正中心，而 `×256` 在解码时使用均匀的箱子边界。只要浮点精度足够（32 位浮点的 23 位尾数远超需求），往返就是无损的。

## 七、最终建议

| 场景 | 推荐方案 |
|------|---------|
| 处理别人的 RGB 图像 | ÷255（标准方案） |
| 自己控制编码和解码 | 可以用 ÷256，但要确保一致 |
| 需要精确零值 | ÷255 |
| 与 GPU 管线交互 | ÷255 |
| 音频处理 | ÷32768（有符号）/ ÷255（无符号） |
| 图像生成（噪声、随机） | ÷256 均匀分布更好 |
| 不确定 | ÷255 |

**一句话总结**：如果你在处理别人给你的图像，除以 255。理论上的优雅不如工程上的一致性重要。但如果你控制整个管线——保存和加载都由你决定——除以 256 的均匀分布在数学上确实更干净。

这个看似微不足道的问题之所以引发激烈讨论，是因为它触及了计算机科学中一个根本性的张力：**理论最优和实践最优往往不是同一个东西**。在浮点与整数的世界里，数学的优雅必须让位于生态系统的惯性。

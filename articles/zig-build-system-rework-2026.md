# Zig 构建系统的双进程革命：从 150ms 到 14ms 的 10 倍提速之路

`zig build --help` 从 150ms 降到 14ms——这不是某个微基准测试的数字游戏，而是 Zig 构建系统架构重写后的实测结果。2026 年 5 月 26 日，Zig 创始人 Andrew Kelley 提交了一个重大分支：将构建系统的"配置"与"执行"拆分为两个独立进程。与此同时，Zig 的自研 ELF 链接器也突破了关键里程碑——能够链接包含 LLVM 和 LLD 库的完整 Zig 编译器，增量重建仅需 30ms。

这两个看似独立的改进，实际上共同指向 Zig 的核心愿景：**让系统级编程的开发体验接近脚本语言的迭代速度**。

## 一、旧架构的瓶颈：一个臃肿的进程做所有事

在 Zig 0.16.0 之前，`zig build` 的工作流程是这样的：

```
build.zig (用户逻辑) + build system (框架代码)
        ↓ 全部编译为一个进程 (Debug 模式)
        ↓ 构建内存中的 build graph
        ↓ 在同一进程中执行 build graph
        ↓ 产出最终产物
```

这个设计有三个根本性问题：

**编译开销无法分摊**。每次运行 `zig build`，即使是微小的参数变更（比如加一个 `-freference-trace`），整个 build.zig 逻辑加上完整的构建系统框架都要重新编译。随着 Zig 构建系统功能不断增加（`--watch`、`--fuzz`、`--webui`），这个"框架代码"的体积持续膨胀，但用户的 build.zig 可能只有几十行。

**执行效率受限**。build graph 的执行器和用户逻辑共享同一个 Debug 编译的进程。Debug 模式意味着没有优化——分支预测、内联、循环展开全部缺失。对于一个每次构建都会调用的执行器来说，这是纯粹的浪费。

**工具链集成困难**。第三方工具如 ZLS（Zig Language Server）需要理解构建系统的内部状态，但旧架构下这些状态被锁在一个不透明的进程内部。

## 二、新架构：Configurer 与 Maker 的分离

Andrew Kelley 的解决方案是将单一进程拆分为两个：

```
build.zig (用户逻辑)
    ↓ 编译为 "configurer" 进程 (Debug 模式)
    ↓ 构建 build graph
    ↓ 序列化为二进制配置文件
    ↓ 缓存（下次可跳过）

build system (框架代码)
    ↓ 编译为 "maker" 进程 (Release 模式，全局缓存)
    ↓ 读取配置文件
    ↓ 执行 build graph
    ↓ 产出最终产物
```

这个设计的精妙之处在于三个关键洞察：

### 2.1 变化的部分与不变的部分

用户的 build.zig 逻辑是频繁变化的——每次调整构建参数、添加依赖、修改编译选项都会改变它。但构建系统的框架代码是相对稳定的，它只在 Zig 版本升级时才需要重新编译。

旧架构把两者捆绑在一起，导致"变"的部分拖累"不变"的部分。新架构让 configurer（变化的）和 maker（不变的）独立编译、独立缓存。maker 进程只需要**每个 Zig 版本编译一次**，之后通过全局缓存复用。

### 2.2 序列化是万能胶水

将 build graph 序列化为二进制配置文件，这一步看似简单，却打开了多扇门：

- **缓存**：如果 build.zig 没变，直接复用上次的配置文件，跳过整个 configurer 阶段
- **工具集成**：ZLS 等工具可以直接解析配置文件，不再需要 fork build runner
- **参数变更免重编译**：加一个 `-freference-trace` 不再触发 build.zig 重编译

### 2.3 Debug vs Release 的正确分配

configurer 用 Debug 模式编译是合理的——它只运行一次来生成配置，编译速度比运行速度更重要。但 maker 是每次构建都会调用的执行器，用 Release 模式编译才能发挥最大性能。

## 三、性能数据：数字说话

Andrew Kelley 在 devlog 中给出了 `zig build --help` 的 benchmark，这组数据值得仔细看：

| 指标 | 旧架构 | 新架构 | 提升 |
|------|--------|--------|------|
| wall_time | 150ms | 14.3ms | **-90.4%** |
| cpu_cycles | 593M | 24.1M | **-95.9%** |
| instructions | 995M | 43.7M | **-95.6%** |
| cache_references | 25.8M | 1.46M | **-94.3%** |
| peak_rss | 84.8MB | 78.5MB | -7.4% |

wall_time 从 150ms 降到 14ms，这是一个质变——从"能感知到的延迟"变成了"几乎瞬间"。更值得注意的是 cpu_cycles 和 instructions 的降幅（95%+），说明这不是某个局部优化，而是架构层面的效率飞跃。

之所以如此 dramatic，是因为旧架构下 `zig build --help` 也要完整执行一遍 build.zig 逻辑，而新架构直接使用缓存的配置文件，跳过了整个 configurer 阶段。

## 四、ELF 链接器：增量编译的最后一块拼图

构建系统提速只是故事的一半。另一半是 Zig 自研 ELF 链接器的突破性进展。

### 4.1 从"只能链接 Zig 代码"到"能链接 LLVM"

Zig 0.16.0 首次引入了新的 ELF 链接器（通过 `-fnew-linker` 启用），但当时它只能处理纯 Zig 代码，连 libc 都不支持。截至 2026 年 5 月 30 日，这个链接器已经能够完成一个极具挑战性的任务：**用 LLVM 和 LLD 库构建 Zig 编译器本身**。

```bash
# 用新链接器构建 Zig 编译器（含 LLVM/LLD）
zig build -Dno-lib -Dnew-linker -Denable-llvm

# 用构建出的编译器编译一个程序
./zig-out/bin/zig build-exe hello.zig -llvm -lld
./hello
# Hello, World!
```

这意味着新链接器已经支持：符号解析、重定位、动态链接、PLT/GOT 处理、TLS、共享库加载等 ELF 链接的全部核心功能。

### 4.2 30ms 增量重建

新链接器的杀手特性是**快速增量编译**。在 Andrew Kelley 的 Tetris 示例中，每次代码修改后的重建只需约 30ms。更惊人的是，这个速度在链接外部 C 库时也没有额外开销。

用 Zig 编译器自身的构建作为测试：

```bash
zig build -Dno-lib -Denable-llvm -fincremental --watch
# 首次构建: 36s
# 后续增量: 244ms, 228ms, 288ms, 283ms
```

从 36 秒到 244 毫秒——**150 倍的加速**。这不是理论数字，而是构建 Zig 编译器这种大型项目的真实数据。

### 4.3 当前状态与下一步

新链接器目前仍默认禁用（需 `-fnew-linker` 启用），最大的缺失功能是不支持生成 DWARF 调试信息。作者 Matthew Lugg 将此列为最高优先级。Windows 的 COFF 链接器也在社区贡献者的推进中。

对于使用 Zig master 分支的 x86_64 Linux 用户，现在就可以尝试增量编译：

```bash
zig build -fincremental --watch
```

## 五、社区反响与生态影响

HN 社区对这两个改进的反应（509pts、266 条评论）揭示了几个有趣的信号：

### 5.1 "这才是我要的编译速度"

一位用户总结得很到位：

> "Zig 的编译速度已经很棒了，这只会让它更好。"

但也有务实的声音指出，初始编译（冷启动）仍然较慢，特别是 ZLS 需要重建时的"令人痛苦的暂停"。增量编译的真正价值在于**热路径**——反复修改-编译-运行的开发循环。

### 5.2 Zig 的定位越来越清晰

社区中反复出现一个类比：

> "如果 Rust 是新的 C++，Zig 就是新的 C。"

Zig 不追求 Rust 式的所有权系统和内存安全保证，而是追求**极致的编译速度和开发体验**。一位用户描述 Zig 是"车库捣鼓"语言——拿起就能用，每次碰壁时发现作者已经想到了解决方案。

### 5.3 与 Bun/Rust 的关系

有评论将 Zig 的进展与 Bun 转向 Rust 的决策联系起来。Zig 核心团队成员 kristoff_it 澄清：

> "这些工作已经进行了很长时间，与 Bun 无关。人们只是因为'戏剧性'才开始关注这些进展。"

实际上，Zig 的去 LLVM 化计划可以追溯到 2020 年——比 Bun 的 Rust 转向早了多年。

### 5.4 作为编译目标的潜力

一位正在开发内存安全语言的开发者表示，选择 Zig 而非 Rust 作为编译目标是正确的决定：

> "Zig 有最好的构建系统设计，是最好的编译目标。"

这指向了 Zig 的一个被低估的价值：作为其他语言后端的"更好的 C"。

## 六、构建系统设计的启示

Zig 的构建系统重写给所有语言的工具链设计者提供了几个有价值的教训：

### 6.1 分离关注点

变化频繁的用户逻辑和稳定的框架代码应该独立编译、独立缓存。这个原则在 Make（Makefile vs make 二进制）、CMake（CMakeLists.txt vs cmake）中都有体现，但 Zig 走得更远——它用序列化配置文件作为中间层，实现了真正的解耦。

### 6.2 缓存粒度很重要

Zig 的 maker 进程按 Zig 版本缓存，这意味着升级 Zig 时才需要重新编译。而 configurer 的输出按 build.zig 内容缓存——如果 build.zig 没变，直接复用上次结果。两层缓存，不同的粒度，覆盖了所有常见场景。

### 6.3 Debug/Release 不是一刀切

传统构建系统通常对所有组件使用相同的编译模式。Zig 的做法是根据组件的实际使用模式选择最优编译策略——只运行一次的用 Debug（快编译），反复运行的用 Release（快执行）。

## 七、未来展望

Zig 0.17.0 即将发布，届时这些改进将进入稳定发布通道。对于犹豫是否要尝试 Zig 的开发者，这可能是一个好的入场时机：

- **构建速度**：增量编译 30ms 级别，接近 Go 的编译体验
- **链接器**：自研 ELF 链接器逐步替代 LLD，消除对外部工具的依赖
- **去 LLVM 化**：Zig 自带的后端持续改进，未来大部分项目可能不再需要 LLVM
- **工具链**：ZLS 通过序列化配置文件获得更好的构建系统集成

Zig 的哲学始终是"更好的 C"——不通过抽象隐藏复杂性，而是通过更好的工具让复杂性变得可控。这次构建系统的重写，完美践行了这一哲学。

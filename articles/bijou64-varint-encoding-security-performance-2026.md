# Bijou64：当安全约束催生出更快的编码——变长整数设计的意外之旅

> "我们本来只是想修一个签名验证的 bug，结果意外得到了一个更快的编码。"

2026 年 4 月，Ink & Switch 的 Brooklyn Zelenka 发布了一个名为 **Bijou64** 的变长整数编码。这个编码原本不是为了性能而设计的——它是为了消除 LEB128 中一个微妙的安全漏洞。但最终的基准测试结果让所有人都吃了一惊：**解码速度比 LEB128 快了 2 到 10 倍**。

这不是又一个"我比你快"的基准测试故事。这是一个关于**设计约束如何意外催生出更优解**的工程叙事。

## 什么是变长整数？为什么你需要关心？

在二进制协议的世界里，有一个永恒的权衡：整数应该占多少字节？

如果你固定用 8 字节（64 位）来存储每个整数，那么 `1` 和 `18,446,744,073,709,551,615` 占用相同的空间。这就像用 10 位数字来写所有的数——写 `5` 要写成 `0000000005`。荒谬，但对齐性好。

**变长整数**（Variable-length Integer，简称 varint）解决这个问题：小数字用少字节，大数字用多字节。这不是什么新技术——DWARF 调试格式用 LEB128，Protocol Buffers 有自己的 zigzag 编码，SQLite 有自定义 varint，QUIC 协议（HTTP/3 的传输层）也定义了变长整数。

但这里有一个大多数人都忽略的问题：**同一个数字可以有多种编码方式**。

## LEB128 的"多重人格"问题

LEB128 是最广泛使用的 varint 编码。它的原理很直观：每个字节的低 7 位存储数据，最高位是"继续位"（continuation bit）——如果为 1，表示"还有后续字节"。

```
数字 300 的 LEB128 编码：
  0x82 0x02
  10000010 00000010
  ↑ 继续位  ↑ 结束
  低7位: 0000010 = 2
  低7位: 0000010 = 2
  值 = 2 + (2 << 7) = 2 + 256 = 300 ✓
```

简单、优雅、被广泛采用。但它有一个致命的特性：

**数字 0 可以被编码为：**
- `0x00`（1 字节）
- `0x80 0x00`（2 字节）
- `0x80 0x80 0x00`（3 字节）
- ...任意长度的 `0x80` 序列后跟 `0x00`

不只是零。**几乎所有数字在 LEB128 中都有多种合法表示**。这不是 bug，是设计的副产品——继续位机制本质上允许"前导零"。

你可能觉得："谁会故意构造恶意的 varint？" 答案是：**任何能从你的协议误判两个不同字节串为同一个值中获益的人。**

## 从 ASN.1 到 PKCS#1：规范化的血腥历史

这不只是理论上的担忧。规范性（canonicality）攻击在密码学中有深厚的"战功"：

**ASN.1 BER 编码**（X.509 证书、LDAP、无数关键基础设施的底层格式）允许整数有前导零字节。这直接导致了：

- **PKCS#1 v1.5 签名伪造**：Bleichenbacher 在 2006 年展示了如何利用 ASN.1 编码中的填充歧义来伪造 SSL 签名。攻击者可以构造一个字节串，让验证器认为它是一个合法的签名，即使它不是。
- **Mozilla NSS 库漏洞**：类似的编码歧义被发现存在于 Firefox 使用的 NSS 加密库中。
- **GnuTLS 规范化失败**：GnuTLS 多次因为 ASN.1 解码中的规范化问题而发布安全更新。
- **JWT 签名绕过**：JSON Web Token 的某些实现中，编码差异导致签名验证被绕过。
- **Bitcoin 交易延展性**：Bitcoin 早期的交易 ID 可以通过修改签名的编码方式（但不改变其语义）来改变，导致了著名的"交易延展性攻击"。

所有这些攻击的模式都是一样的：

1. 数据结构允许多种编码表示同一个值
2. 签名/哈希作用于原始字节
3. 验证器将字节解码为语义值后比较
4. 攻击者构造一个字节不同但语义相同的编码
5. 签名验证通过（因为语义相同），但哈希不同（因为字节不同）

**如果你的协议需要对数据做签名、哈希、或任何依赖字节级一致性的操作，非唯一编码就是一个定时炸弹。**

## Bijou64 的设计哲学：让正确性成为唯一选项

大多数 varint 的解决方案是：先用你喜欢的编码，然后在解码后加一个"这是规范形式吗？"的检查。Bijou64 的方案不同：**让非规范编码在结构上不可能存在**。

就像我们日常使用的十进制数字系统——`42` 就是 `42`，没有另一种写法。

Bijou64 用了两个巧妙的设计：

### 第一个技巧：标签字节（Tag Byte）

第一个字节的值决定了一切：

- **0x00 – 0xF7（0–247）**：这个字节本身就是完整的数字。1 字节搞定。
- **0xF8 – 0xFF（248–255）**：这是一个"标签"，告诉你后面跟着多少字节。具体来说，标签值 `T` 意味着后面有 `T - 0xF7` 个数据字节。

```
0xF8 → 后面跟 1 字节（共 2 字节）
0xF9 → 后面跟 2 字节（共 3 字节）
...
0xFF → 后面跟 8 字节（共 9 字节）
```

**关键优势**：解码器读到第一个字节后，**立即知道**总共需要读多少字节。这是 O(1) 的。而 LEB128 必须逐字节扫描继续位，直到找到结束标记——这是 O(n) 的。

### 第二个技巧：偏移消除（Offset Elimination）

光有标签还不够——你还需要确保 `0xF8 0x00` 不等于 `0x00`（否则又回到了多重表示的问题）。

Bijou64 的解决方案是：**为每个长度级别设置一个基础偏移值**。

```
1 字节: 直接表示 0–247，偏移 = 0
2 字节: 值 = 第二字节 + 248，偏移 = 248
3 字节: 值 = 后两字节大端值 + 504，偏移 = 504
4 字节: 偏移 = 66,040
...
```

偏移的规律是：`OFFSET[n] = OFFSET[n-1] + 256^(n-1)`。每个级别的编码范围恰好与上一级不重叠。

### 解码实例：1738 是怎么编码的？

让我们跟着解码器走一遍：

```
字节流: 0xF9 0x06 0x3A

第 1 步: 读第一字节 0xF9 = 249
         249 ≥ 248，进入"标签模式"
         标签 249 意味着后面有 249 - 247 = 2 字节

第 2 步: 读后面 2 字节，大端解释
         0x06 0x03A = 6 × 256 + 58 = 1594

第 3 步: 加上 3 字节级别的偏移
         OFFSET[3] = 248 + 256 = 504
         最终值 = 1594 + 504 = 1738 ✓
```

**注意**：对于 1738 这个数字，LEB128 也需要 2 字节（7 位 × 2 = 14 位，足够表示 16384 以下的数字）。但 LEB128 的 2 字节范围是 128–16,383，而 Bijou64 的 2 字节范围是 248–65,791。Bijou64 在 2 字节能表示更大的数字。

## 边界情况：9 字节的特殊处理

有一个例外需要手动检查。9 字节（标签 + 8 字节数据）的理论范围超过了 `u64` 的最大值（因为偏移的存在）。由于 Bijou64 的目标是无符号 64 位整数，当标签为 `0xFF` 时，解码器需要额外检查值是否小于 `2^64`。

这不是规范性问题——每个合法范围内的数字仍然只有一种表示——只是在裁剪不需要的额外空间。

## 基准测试：意外的胜利

Ink & Switch 在两个平台上做了基准测试：

- **ARM**：Apple M2 Pro
- **x86**：AMD Zen 5

### 解码性能

| 数字分布 | Bijou64 | LEB128 | 倍数 |
|---------|---------|--------|------|
| 小数字（0–247） | ~0.5 ns/值 | ~1 ns/值 | **2×** |
| 中等数字（248–65535） | ~1 ns/值 | ~3 ns/值 | **3×** |
| 大数字（>65535） | ~1.5 ns/值 | ~12 ns/值 | **8×** |
| 均匀 u64 分布 | ~0.75 ns/值 | ~7.3 ns/值 | **~10×** |

在最"对抗性"的基准测试（均匀分布在完整 u64 范围内）中，Bijou64 处理 4096 个值仅需约 3 微秒，而 LEB128 需要约 30 微秒。

### 方差故事：CDF 曲线揭示的真相

基准测试的中位数只是一半的故事。**累积分布函数**（CDF）曲线揭示了更深层的差异：

- **Bijou64 的 CDF 曲线几乎是垂直的**——每个批次的计时都紧密聚集在中位数附近。
- **LEB128 的 CDF 曲线向右倾斜拖尾**——因为继续位的扫描长度取决于数值本身，分支预测器永远无法稳定下来。

这意味着 Bijou64 不仅更快，而且**更可预测**。在实时系统和低延迟场景中，可预测性往往比平均速度更重要。

### 编码性能

编码方面，Bijou64 在大多数分布下也更快。唯一的例外是 248–65,535 范围的数字，LEB128 以约 1.24 倍的优势略胜。

### 规范解码：免费的午餐

最令人惊讶的是规范性解码的性能对比。由于 Bijou64 的规范性是结构保证的，**规范解码就是普通解码**。而 LEB128 的规范解码需要额外的验证步骤：

```
Bijou64 规范解码 = 普通解码（0 额外开销）
LEB128 规范解码 = 普通解码 + 逐字节规范化检查（显著额外开销）
```

如果你的场景需要签名验证、内容寻址、或任何依赖字节级一致性的操作，Bijou64 的优势会被进一步放大。

## 为什么更快？架构层面的解释

Bijou64 更快不是因为"更聪明的实现"，而是因为**它需要做的工作更少**：

### 1. 无继续位扫描

LEB128 的解码器必须检查每个字节的最高位，决定是否继续读取。这是一个**数据依赖的分支**——分支预测器无法提前知道下一个字节是否是最后一个。

Bijou64 从第一个字节就知道总共要读多少字节。这是一个**固定模式的分支**——分支预测器很快就能学会。

```
LEB128 解码循环：
while (byte & 0x80):  // 数据依赖分支，预测器讨厌这个
    byte = next_byte()
    value |= (byte & 0x7f) << shift
    shift += 7

Bijou64 解码：
tag = first_byte
count = tag_to_count[tag]  // 查表，O(1)
bytes = read_n_bytes(count)  // 精确读取
value = big_endian_decode(bytes) + offset[tag]  // 一条 bswap 指令
```

### 2. 大端整数载荷

Bijou64 的数据部分是一个连续的大端整数，不是 LEB128 那种分散的 7 位块。现代 CPU 有专用的字节交换指令（`bswap`），编译器会将读取优化为**一次加载 + 一次字节交换**。

LEB128 的 7-bit-per-byte 布局迫使解码器对每个字节做掩码和移位操作。

### 3. 查表 + 加法代替分支

Bijou64 的偏移计算用的是一个小查找表 + 一次加法。之前的版本用的是 `if` 分支，但算术版本在大多数现代 CPU 上的热路径中指令更少。

### 4. 分支预测友好

对于任何特定的工作负载，数字的大小分布通常是稳定的。Bijou64 的标签匹配模式让分支预测器几乎立刻就能稳定下来——这正是那些陡峭的 CDF 曲线所展示的。

## 与其他 Varint 编码的对比

| 编码 | 规范性 | 最大范围 | 解码复杂度 | 典型场景 |
|------|--------|---------|-----------|---------|
| **LEB128** | 非结构保证 | 任意精度 | O(n) 扫描 | DWARF, Wasm |
| **Protobuf Varint** | 非结构保证 | 64 位 | O(n) 扫描 | gRPC, Protobuf |
| **SQLite Varint** | 类似思路 | 64 位 | O(1) 查表 | SQLite 内部 |
| **QUIC Varint** | 结构保证 | 62 位 | O(1) | HTTP/3 |
| **Bijou64** | 结构保证 | 64 位 | O(1) 查表 | CRDT, 签名协议 |

**SQLite 的 varint** 值得一提。SQLite 使用了一种与 Bijou64 思路相似的编码：第一个字节的高位模式决定了后续字节数。HN 评论者 `conaclos` 指出了两者的相似性。区别在于 SQLite 的设计更保守，没有做偏移优化，因此在紧凑性上不如 Bijou64。

**QUIC 的变长整数** 也采用了结构保证规范性的设计（2 位长度前缀 + 数据），但最大只支持 62 位，且设计目标不同（网络包头编码，强调简单性）。

**Elias omega coding**（HN 评论者 `yread` 提到）来自信息论领域，是一种递归自描述编码。理论上很优雅，但在实际的二进制协议中过于复杂。

## 实际应用场景

### CRDT 同步协议

Bijou64 最初是为 Ink & Switch 的 **Subduction CRDT 同步协议**开发的。在 CRDT（Conflict-free Replicated Data Type）系统中，多个节点独立修改数据然后同步。同步过程需要对数据做签名和哈希，因此字节级一致性至关重要。

```
节点 A: 编码数字 42 → 字节 [0x2A]
节点 B: 编码数字 42 → 字节 [0x2A]  ← 必须相同！

如果用 LEB128:
节点 A: 编码数字 42 → 字节 [0x2A]
节点 B: 编码数字 42 → 字节 [0x80, 0x2A]  ← 不同！签名验证失败
```

### 内容寻址存储

在 content-addressed 系统（如 IPFS、Git）中，数据的哈希就是它的地址。如果同一个逻辑值可以有不同的字节表示，它们就会有不同的哈希——导致"相同内容，不同地址"的混乱。

### 安全关键协议

任何需要对结构化数据做签名的协议（TLS、Signal Protocol、各种区块链协议）都能从结构保证的规范性中受益。

### 高吞吐量数据管线

即使你不需要规范性，2–10 倍的解码速度提升在高吞吐量场景中也很有价值：日志处理、时序数据库、消息队列序列化。

## 局限性与注意事项

1. **新格式，缺乏实战验证**。LEB128 有数十年的成熟实现，Bijou64 发布于 2026 年 4 月，测试覆盖的 CPU 型号有限（M2 Pro、Zen 5、Zen 3）。

2. **最大 64 位**。LEB128 和 Protobuf Varint 可以处理任意精度的整数。如果你需要表示超过 `2^64` 的数字，Bijou64 不适用。

3. **SIMD 优化困难**。HN 评论者 `kstenerud` 指出，这种基于首字节标签的方案难以利用 SIMD 指令进行批量解码。对于极端吞吐量需求，可能需要不同的方案。

4. **编码大小在某些分布下不如 LEB128**。LEB128 在 128–16,383 范围内始终保持 2 字节，而 Bijou64 的 2 字节范围是 248–65,791。如果你的数据集中在 128–247 范围，LEB128 更紧凑。

5. **生态成熟度**。目前只有 Rust crate（`bijou64`，约 3,200 次下载）和一个 Wasm/JavaScript 封装。`bijou32` 和 `bijou128` 还在规划中。

## 如何使用

### Rust

```toml
# Cargo.toml
[dependencies]
bijou64 = "0.1"
```

```rust
use bijou64::{encode, decode};

fn main() {
    // 编码
    let mut buf = [0u8; 9];
    let n = encode(1738, &mut buf);
    println!("1738 编码为 {} 字节: {:?}", n, &buf[..n]);
    // 输出: 1738 编码为 3 字节: [0xF9, 0x06, 0x3A]

    // 解码
    let (value, consumed) = decode(&buf[..n]).unwrap();
    println!("解码得到: {} (消费了 {} 字节)", value, consumed);
    // 输出: 解码得到: 1738 (消费了 3 字节)
}
```

### Wasm/JavaScript

```javascript
// 通过 Wasm 封装使用
import init, { encode, decode } from 'bijou64-wasm';

await init();
const encoded = encode(1738n); // BigInt
const decoded = decode(encoded);
console.log(decoded); // 1738n
```

## 设计启示：安全约束作为创新催化剂

Bijou64 最有趣的地方不是它的性能数字——而是它如何到达那里。

Zelenka 的团队没有从"让我们做一个更快的 varint"开始。他们从"我们需要一个签名安全的 varint"开始。**规范性的约束迫使他们放弃了继续位机制**，而继续位恰好是 LEB128 性能瓶颈的根源。

这符合一个更广泛的工程模式：**当你为正确性（correctness）而非性能做设计时，你经常会意外地得到性能**。因为正确性的约束消除了"需要额外检查"的情况，而"额外检查"往往就是性能杀手。

其他类似的例子：

- **Rust 的所有权系统**消除了垃圾回收的需要，同时消除了 GC 暂停——最初是为了内存安全，但得到了性能。
- **SQL 的声明式查询**让优化器自由选择执行计划——最初是为了正确表达意图，但常常比手写循环更快。
- **结构化并发**消除了线程泄漏的可能——最初是为了正确性，但简化了资源管理，间接提升了性能。

## 总结

Bijou64 是一个教科书级别的案例研究，展示了：

1. **规范性不是可选的**。如果你的协议涉及签名、哈希或内容寻址，非唯一编码是一个安全漏洞。
2. **好的设计约束可以催生更优的解**。放弃继续位机制（为了安全）反而得到了更好的性能。
3. **O(1) 预知长度 > O(n) 扫描**。这个原则在许多系统设计中都适用。
4. **基准测试要看方差，不只是中位数**。CDF 曲线揭示了分支预测器的喜怒哀乐。
5. **新技术需要时间验证**。在关键路径上替换 LEB128 之前，先在非关键路径上试验。

如果你正在设计一个新的二进制协议、CRDT 同步方案、或任何需要字节级一致性的系统，Bijou64 值得认真考虑。它不只是"另一个 varint"——它是一个关于**设计约束如何引导出更优解**的活生生的证明。
